# Confidence Ledger 设计规约（v0.1）

> 配套 v0.0 spec 中"v0.1 路线图预览"的展开。
> 解决问题：知识库随时间退化、新旧矛盾共存、旧 claim 不会被自动 supersede。
> 状态：v0.0 不实施；v0.1 进入 P0。当前已启动第一步：`packages/confidence` 包含事件 schema、append-only JSONL 读写、基础 score materialization 和 SQLite projection；CLI 已有 `akb migrate to-v0.1`、`akb confidence show`、`akb verify`、`akb supersede` 和 `akb projection rebuild --confidence` 的最小实现；`packages/ranker` 已开始把 confidence state 接进 CLI/MCP 搜索结果排序和 flags。
> 前置依赖：v0.0 的 Page 模型、git-store、patch workflow（v0.1 引入）、v0.0 spec § 0.6 的投影层原则（markdown 是 canonical truth，ledger 的 SQLite 表是投影）。

---

## 0. 为什么必须有

v0.0 的 Page 模型有一个 `confidence: number` 字段但没有计算规则。这是个定时炸弹。**三个具体退化路径**：

### 退化 1：claim 永远新鲜

```yaml
# page_xxx.md，2026-01 创建，2026-01 last_verified
content: "我们使用 Redis 做缓存。"
```

2026-06 团队迁移到 KeyDB。没有人手动更新这页。半年后 LLM 通过 MCP 查到这页，自信地回答"用 Redis"——agent 拿着这个错误信息去改代码。**没有 confidence 衰减就没有自动腐烂检测**。

### 退化 2：矛盾静默共存

```yaml
# page_a.md（2026-01）
content: "GC 触发阈值是 free block < 10%。"

# page_b.md（2026-04，新论文 ingest 后生成）
content: "实验显示 free block < 5% 才触发 GC 更优。"
```

两页都存在，检索时按相关度排序。LLM 同时看到，要么犹豫要么按 token 顺序选一个。**没有 supersession 链就没有"哪个是当前认知"**。

### 退化 3：低质量来源污染

```
akb ingest url <random-blog-post>
↓
生成 10 个新 entity 页，每个 confidence 默认 1.0
↓
跟你 3 年前 FAST paper 总结的 page 同等权重排序
```

没有来源权威性区分，**所有 ingest 都被等同对待**——这是 RAG 系统在生产里最常见的失败模式。

## 1. 设计目标

1. **每个 claim 都能回答**："基于什么信息？最后被确认是什么时候？现在还可信吗？"
2. **新信息不覆盖旧信息**，只新增"事件"。
3. **历史可回放**：删掉任意时间点之后的事件，confidence 应该能从更早的状态重新算出来。
4. **零额外手工维护**：confidence 由 ingest、verify、supersede 等操作自动产生事件，不需要人填字段。
5. **可解释**：每个 `confidence: 0.62` 都能展开成"3 个 source 支持，1 个 contradiction，距上次 verify 73 天，衰减 0.18"。

## 2. 核心抽象：Confidence Ledger

不是一个字段，是一个 **append-only 事件流**。Page 上的 `confidence` 是这个事件流的物化视图。

类比：git 是 file 状态的 append-only commit log；ConfidenceLedger 是 claim 状态的 append-only event log。

### 2.1 事件类型

```typescript
type ConfidenceEvent =
  | { kind: "source_added";     pageId: PageId; sourceId: SourceId; sourceWeight: number; }
  | { kind: "source_removed";   pageId: PageId; sourceId: SourceId; reason: string; }
  | { kind: "verified";         pageId: PageId; verifierType: "human" | "agent"; verifierId?: string; }
  | { kind: "contradicted_by";  pageId: PageId; bySourceId: SourceId; severity: "minor" | "major"; }
  | { kind: "superseded_by";    pageId: PageId; supersederPageId: PageId; }
  | { kind: "supersedes";       pageId: PageId; supersededPageId: PageId; }
  | { kind: "decay_checkpoint"; pageId: PageId; daysSinceLastEvent: number; appliedDecay: number; }
  | { kind: "manual_override";  pageId: PageId; reason: string; newBase: number; };
```

每个事件 base 字段：

```typescript
interface ConfidenceEventBase {
  id: string;                  // ULID, sortable by time
  pageId: PageId;
  timestamp: string;           // ISO 8601
  actor: "human" | "agent" | "system";
  actorId?: string;            // user email, agent name, or "akb-verify-cron"
}
```

### 2.2 物理存储

两层存储：

**a. Markdown 落盘（人类可读 + git 可追）**

每个 page 同目录下生成 `.<page_id>.ledger.jsonl`：

```jsonl
{"id":"01JN...","kind":"source_added","pageId":"page_gc001","sourceId":"src_001","sourceWeight":0.9,"timestamp":"2026-01-15T10:00:00Z","actor":"agent","actorId":"akb-ingest"}
{"id":"01JN...","kind":"verified","pageId":"page_gc001","verifierType":"human","verifierId":"alvin@example.com","timestamp":"2026-02-01T09:00:00Z","actor":"human"}
{"id":"01JN...","kind":"contradicted_by","pageId":"page_gc001","bySourceId":"src_007","severity":"minor","timestamp":"2026-03-22T14:30:00Z","actor":"agent","actorId":"akb-ingest"}
```

JSONL 每行一个事件。append-only，git 友好（每个事件一行 diff）。文件名用点号前缀避免 Obsidian 显示。

**b. SQLite 物化视图（查询用）**

```sql
CREATE TABLE confidence_events (
    id              TEXT PRIMARY KEY,
    page_id         TEXT NOT NULL,
    kind            TEXT NOT NULL,
    timestamp       TEXT NOT NULL,
    actor           TEXT NOT NULL,
    actor_id        TEXT,
    payload         TEXT NOT NULL,        -- JSON of kind-specific fields
    FOREIGN KEY (page_id) REFERENCES pages(id) ON DELETE CASCADE
);

CREATE INDEX idx_conf_events_page_time ON confidence_events(page_id, timestamp);

CREATE TABLE confidence_state (
    page_id              TEXT PRIMARY KEY,
    score                REAL NOT NULL,         -- 0.0 to 1.0
    source_count         INTEGER NOT NULL,
    contradiction_count  INTEGER NOT NULL,
    last_verified_at     TEXT,
    last_event_at        TEXT NOT NULL,
    superseded_by        TEXT,                   -- page_id or NULL
    computed_at          TEXT NOT NULL,
    FOREIGN KEY (page_id) REFERENCES pages(id) ON DELETE CASCADE,
    FOREIGN KEY (superseded_by) REFERENCES pages(id)
);
```

`confidence_state` 是从 `confidence_events` 算出来的缓存。当前实现由 `akb projection rebuild --confidence` 从 canonical JSONL 全量重建这两张表；CLI/MCP search 优先读取 SQLite projection，缺失时回退到逐页读取 JSONL。

### 2.3 哪一份是 source of truth

照 v0.0 spec § 0.6 的投影层原则，**JSONL 文件是 canonical truth，SQLite 是投影**。具体含义：

* **`.ledger.jsonl` 入 git**——事件流是事实，需要被追踪、被 diff、被 PR review
* **SQLite `confidence_events` 和 `confidence_state` 都不入 git**——可以从 JSONL 完全重建
* **冲突时 JSONL 赢**——`akb projection rebuild --confidence` 永远是从 JSONL 重新算出 SQLite，永远不是反过来
* **写入路径**：先 append JSONL → fsync → 写 SQLite events 表 → 重算 state。中间 crash 不一致时，`akb projection rebuild --confidence` 修复
* **手动编辑 JSONL** 是被允许的（紧急修复 / migration）——重建投影后就能生效；这不是 anti-pattern，是 escape hatch

```bash
# 如果 .akb/index.db 损坏或要换机器：
git pull
akb projection rebuild --confidence   # 从 markdown + JSONL 重建 confidence 投影
# 完成，所有 confidence state 都回来了
```

这一条让整套系统跟 markdown 的处理逻辑完全一致，没有"事件流是数据库行"这种 ORM 思维污染。

## 3. Confidence 计算规则

### 3.1 基础公式

```
score(page) = clamp01(
  base
  + source_strength
  - contradiction_penalty
  - time_decay
  + verification_boost
)
```

每一项的具体计算如下。

### 3.2 Source strength（多来源加强）

```
source_strength = 1 - exp(-Σ source_weight_i / k)
```

* `source_weight_i` 是单个 source 的权重（见 3.6）
* `k` 是平滑因子，默认 1.5
* 一个 weight=1.0 的 source → +0.49
* 三个 weight=1.0 的 source → +0.86
* 十个 weight=1.0 的 source → +0.99

含义：来源越多越自信，但收益递减，不会单纯堆数量超过 1.0。

### 3.3 Contradiction penalty（矛盾削弱）

```
contradiction_penalty = min(0.6, Σ severity_weight_i)
  where severity_weight = { minor: 0.10, major: 0.35 }
```

* 一个 minor contradiction → -0.10
* 一个 major contradiction → -0.35
* 多个累加，但上限 0.6（避免单条 claim 直接被打到 0）

### 3.4 Time decay（时间衰减）

参考 Ebbinghaus，但分类型给不同半衰期：

```
days_since = (now - last_event_at).days
time_decay = (1 - 2^(-days_since / half_life)) * decay_ceiling
```

`half_life` 按 page type 区分：

| page type        | half_life (days) | decay_ceiling |
| ---------------- | ---------------- | ------------- |
| `decision` (ADR) | 365              | 0.15          |
| `architecture`   | 180              | 0.20          |
| `module`         | 90               | 0.30          |
| `concept`        | 730              | 0.10          |
| `runbook`        | 60               | 0.40          |
| `note`           | 120              | 0.30          |
| `meeting`        | 30               | 0.50          |
| `api`            | 60               | 0.40          |

含义：架构决策衰减慢（重大决定一年内还基本可信），runbook 衰减快（操作步骤经常变）。每种类型有 ceiling，所以再老也不会衰到 0——还是要看 contradiction。

任何 `verified` 事件**重置 last_event_at**，相当于 Ebbinghaus 的"复习一次曲线回升"。

### 3.5 Verification boost

```
verification_boost = min(0.25, recent_verifications_count * 0.05)
  where "recent" = last 30 days
```

* 一次 verify → +0.05
* 五次 → +0.25（上限）

这是奖励"经常被确认"的 page，比如热门的 runbook。

### 3.6 Source weight（来源权重）

ingest 时根据 source 类型给初始权重：

| source type                | weight |
| -------------------------- | ------ |
| `markdown`（人工写）        | 1.0    |
| `git_commit` (own repo)    | 0.9    |
| `code` (parsed by code-intel) | 0.9 |
| `github_pr` (merged)       | 0.8    |
| `github_issue` (closed)    | 0.6    |
| `meeting` (transcribed)    | 0.7    |
| `pdf` (academic)           | 0.8    |
| `pdf` (vendor whitepaper)  | 0.5    |
| `webpage` (known authority) | 0.6   |
| `webpage` (unknown)        | 0.3    |
| `chat`                     | 0.4    |

"known authority" 是 config 里的 allowlist：

```yaml
# .akb/config.yaml
sources:
  authority_domains:
    - "*.usenix.org"
    - "*.acm.org"
    - "docs.anthropic.com"
    - "github.com/anthropics/*"
```

Allowlist 命中 → +0.3 to default。

### 3.7 Confidence-aware retrieval（confidence 进入检索 score）

到这里 confidence 还只是 page 的一个属性。要让它真正影响 agent 看到的东西，必须**进 retrieval ranking**。否则 confidence 仅仅是 metadata，agent 拿到一堆 chunk 时还是按 BM25 排，看不见可信度差异。

#### 设计目标

* 高相关 + 高 confidence → 排第一（理想结果）
* 高相关 + 低 confidence → 仍然出现在结果里，但**标记 NEEDS_REVIEW**，不藏起来
* 低相关 + 高 confidence → 不强行拉高（相关性还是主信号）
* 被 supersede 的 page → 默认不出现，除非显式 `--include-superseded`

**反模式**：乘法 `score × confidence` 会把 confidence=0.3 的 page 直接压到接近 0，等于隐性删除——reviewer 永远看不到、永远没机会决定是 supersede 还是 re-verify。这跟 ledger "标记腐烂、不偷偷藏"的哲学冲突。**所以用加权和，不用乘法**。

#### 公式

```
final_score = bm25_normalized * w_relevance
            + confidence       * w_confidence
            + freshness        * w_freshness
            + access_recency   * w_access

where:
  bm25_normalized = 1 / (1 + bm25_raw)            // FTS5 返回越小越好，先反转
  confidence      = page.score from ledger         // 已经 0-1
  freshness       = exp(-days_since_update / 180)  // 半年半衰
  access_recency  = exp(-days_since_last_query / 30)

default weights (sum to 1.0):
  w_relevance   = 0.55
  w_confidence  = 0.25
  w_freshness   = 0.10
  w_access      = 0.10
```

权重在 `.akb/config.yaml` 可调，但默认值经过实际 eval 锁定，调之前必须先跑 golden set 验证不退化。

#### 实现位置

不在 `search-engine` 包里硬编码——`search-engine` 只算 BM25 和原始 chunk，**重排在一个独立的 `packages/ranker` 包里**做。当前最小实现已经新增 `packages/ranker`，并在 CLI `akb search` 与 MCP `search_knowledge` 中优先使用 SQLite `confidence_state`，缺失时回退 ledger-derived confidence state。

```typescript
// packages/ranker
interface RankInput {
  rawResults: SearchResult[];     // from search-engine
  confidenceState: Map<PageId, ConfidenceState>;
  pageAccessLog?: Map<PageId, Date>;  // optional, last-queried timestamps
  weights?: RankWeights;
  options?: { includeSuperseded?: boolean };
}

interface RankedResult extends SearchResult {
  final_score: number;
  component_scores: {
    relevance: number;
    confidence: number;
    freshness: number;
    access_recency: number;
  };
  flags: ResultFlag[];   // e.g. ["NEEDS_REVIEW", "RECENTLY_CONTRADICTED"]
}
```

`flags` 通过 MCP 返回给 agent，让 LLM 自己看到"这页相关但置信度低，需要 cross-check"。MCP tool schema 里要明确告诉 LLM 这个字段的含义。

#### 边界情况

* **被 supersede 的 page**：默认在 SQL 层 filter（`WHERE superseded_by IS NULL`），不进 ranker。`include_superseded=true` 时返回，且 finalScore 额外 ×0.5
* **完全没有 ledger 数据的 page**（v0.0 兼容 / 新 ingest 还没事件）：`confidence` 默认 0.7（中性偏正），`access_recency` 默认 0.5
* **score 全为 0 的 chunk**：跳过（BM25 没匹配上）
* **flags 触发阈值**：`confidence < 0.5 → NEEDS_REVIEW`；`最近 30 天有 major contradiction → RECENTLY_CONTRADICTED`；`last_verified_at > 半衰期 × 2 → STALE`
* **当前实现边界**：已支持 `NEEDS_REVIEW`、`STALE`、`SUPERSEDED`，并默认过滤 superseded pages；`RECENTLY_CONTRADICTED` 需要 ledger materialized projection 或事件摘要，尚未实现。

#### 跟 v0.0 的关系

v0.0 没有 confidence 也没有 ranker——`search` 直接返回 bm25 排序的 raw 结果。v0.1 引入时**保持 search-engine 的 API 完全不变**，只在它上面加一层 ranker。这样 v0.0 build 出来的所有东西都不需要改。

## 4. Supersession（取代链）

矛盾消解的核心机制。**不删除旧 page，建立"已被取代"链**。

### 4.1 触发场景

ingest 一个新 source 时，LLM compile 阶段判定它跟某个现有 page **直接矛盾**（不只是 minor 差异），会生成 supersede 提议：

```
[Patch Proposal]
  - Update page_gc001 (current confidence 0.71)
  - OR create page_gc002 that supersedes page_gc001

  New source: src_007 "FAST 2026 paper: adaptive GC thresholds"
  Conflict: claims threshold should be 5%, page_gc001 says 10%

  Recommendation: supersede (substantial methodology change)
```

人或 agent review patch，决定 update（小改）还是 supersede（大变）。

### 4.2 链的语义

```
page_gc001 (2026-01)  ← original
   ↑ superseded_by
page_gc002 (2026-04)  ← new version, references gc001 as historical context
   ↑ superseded_by
page_gc003 (2026-09)  ← further revision
```

* 检索默认**只命中链尾**（`superseded_by IS NULL`）
* `--include-superseded` 标志才返回历史版本
* 链尾 page 在 frontmatter 里显式 link 上一版：

```yaml
---
id: page_gc003
supersedes: page_gc002
---

> Supersedes [[page_gc002]], which superseded [[page_gc001]].
> Historical reasoning preserved in those pages.
```

* 链中间的 page 不可再被新 supersede（防止分叉）；要分叉必须先 `akb supersede --unlink` 显式断开

### 4.3 confidence 在链中的传播

* 被 supersede 的 page，confidence 立即 cap 在 0.3（降级到"历史参考"）
* supersede 关系本身是一个 source（`superseded_by` 事件），不直接 boost 新 page 的 confidence
* 检索 reranker 在看到 supersede 链时给新 page +0.1 boost（信号：这是"经过迭代"的 claim）

## 5. CLI 命令

### 5.1 `akb verify <page-or-glob>`

人工或定期 verify。生成 `verified` 事件。

```bash
akb verify page_gc001
akb verify "pages/runbooks/**"
akb verify --by-agent claude-code --reason "agent ran the runbook successfully"
```

当前最小实现支持 page id、page path、`pages/**` glob、`--by-agent`、`--reason`、`--dry-run` 和 `--no-commit`。`--dry-run` 使用 0.70 作为低 confidence 阈值，只报告、不写 ledger。

可被 CI 调用：

```yaml
# .github/workflows/verify.yml
- name: Verify runbooks against current code
  run: akb verify "pages/runbooks/**" --by-agent ci --dry-run
```

`--dry-run` 模式：只输出哪些 page 的 confidence < 阈值，不写事件。

### 5.2 `akb confidence show <page>`

诊断命令。展开当前 score 的来源：

```
$ akb confidence show page_gc001
$ akb confidence show page_gc001 --format json

page_gc001  "Garbage Collection Strategy"
  current score: 0.62 (down from 0.84 at last verify)

  breakdown:
    base                  +0.50
    source_strength       +0.79  (3 sources: src_001 w=1.0, src_004 w=0.9, src_007 w=0.5)
    contradiction_penalty -0.10  (1 minor contradiction from src_007)
    time_decay            -0.27  (73 days since last_event, type=module, half_life=90)
    verification_boost    +0.00

  events: 8 total
    2026-01-15  source_added       src_001 (w=1.0)
    2026-01-15  source_added       src_004 (w=0.9)
    2026-02-01  verified           alvin@example.com
    2026-03-22  contradicted_by    src_007 (severity=minor)
    2026-03-22  source_added       src_007 (w=0.5)
    2026-03-22  decay_checkpoint   45 days, -0.10
    2026-05-13  decay_checkpoint   52 days, -0.17
    2026-05-13  manual_override    none

  status: NEEDS_REVIEW
    reason: score < 0.7 and last verified > 60 days ago
```

### 5.3 `akb supersede <old> --by <new>`

显式建立 supersede 关系。

```bash
akb supersede page_gc001 --by page_gc002 \
  --reason "FAST 2026 paper supersedes original threshold model"
```

行为：
1. 写 `superseded_by` 事件到 old page
2. 写 `supersedes` 事件到 new page
3. 更新 new page frontmatter 加 `supersedes: page_gc001`
4. 在 new page 顶部插入 `> Supersedes [[page_gc001]].`
5. git commit

当前最小实现支持 page id 或 page path 作为 old/new 参数，`--reason` 会进入两边 ledger 事件，`--no-commit` 可跳过 git commit。命令会刷新 new page 的 FTS 投影。

### 5.4 `akb lint`

完整健康检查（v0.0 已有 verify，v0.1 扩展）：

```
$ akb lint

Scanning 1,243 pages...

Confidence issues:
  ⚠  47 pages with score < 0.5 (see .akb/lint/low-confidence.md)
  ⚠  12 pages last verified > 180 days ago
  ✗  3 pages with conflicting active claims (unresolved contradictions)

Structural issues:
  ⚠  8 orphan pages (no incoming or outgoing links)
  ⚠  5 broken wiki links
  ✗  2 supersession cycles detected

Suggestions:
  📌 page_gc001 (score 0.42): consider supersede or re-verify
  📌 page_meeting_042 (no source citations): downgrade to "note" type?
```

Exit code 1 if any ✗ (CI gate).

### 5.5 `akb decay --run`

后台衰减计算。一般由 cron 或 git hook 触发，不让用户手动跑：

```bash
# 每天跑一次，给所有 page 加 decay_checkpoint 事件
akb decay --run --since-last 24h
```

实现注意：**不每天都加事件**。只有当上次 decay event 距今超过 14 天，或 score 跨过阈值（0.7 / 0.5 / 0.3）时才写事件。否则 ledger 会被 decay checkpoint 噪音淹没。

### 5.6 Runtime verification（系统行为隐式验证）

`akb verify` 是显式触发的，**runtime verification 是从真实系统行为自动产生 `verified` 事件**——这是 confidence ledger 真正闭环的关键。

#### 5.6.1 为什么需要

显式 verify 有结构性缺陷：

* 人懒得跑——大部分 page 永远不会被手动 verify
* agent 自己说 "verified" 是循环论证——LLM 给自己写的东西打"已验证"没有信息量
* CI 跑通过本身就是对 page 的隐式确认——为什么不用？

真实信号：

* **runbook 执行成功** 且 page 描述了执行步骤 → page 描述还有效
* **CI 通过** 且 PR 修改了 page 引用的代码 → 代码与文档一致
* **integration test 通过** 且 test 名匹配 page id 或 page tag → 行为契约满足
* **benchmark 跑出预期数字** 且 page 写了 perf 预期 → 性能 claim 还成立
* **deploy succeeded** 且 page 是 deploy runbook → runbook 步骤可执行

#### 5.6.2 进入方式

三条路径，按强度排：

**a. 显式 hook 命令（强）**

```bash
# Runbook 执行
akb runbook exec page_runbook_gc_tuning
# → 跑 page 里 fenced code block，全部 0 退出码就写 verified
# 失败也写事件：{ kind: "runbook_failed", failedStep: 3, error: "..." }

# 关联测试
akb test --link-pages
# → 跑 `pnpm test`，对每个测试解析 @akb-page <id> 标注（test 注释里的）
# → 测试通过 → page 写 verified；失败 → 写 contradicted_by self
```

**b. CI Webhook（中）**

```bash
# 在 .github/workflows/main.yml 末尾
- name: Notify akb
  if: success()
  run: |
    akb webhook ci-success \
      --pr-number ${{ github.event.pull_request.number }} \
      --changed-files-list .pr-changed-files.txt
```

`akb webhook ci-success`：
1. 读 changed files
2. 找出哪些 page frontmatter 里 `references:` 包含这些文件
3. 对每个匹配 page 写 `verified` 事件，actor=`ci:github-actions`

**c. 文件系统观察（弱）**

```bash
# 长期跑的守护进程，watch 项目里的 ./.akb/runtime-signals/
akb watch
```

外部脚本 / make target / CI 步骤往 `.akb/runtime-signals/<timestamp>.json` 丢信号：

```json
{ "kind": "deploy_success", "page_ids": ["page_runbook_prod_deploy"], "actor_id": "deploy-bot" }
```

`akb watch` 消费这些文件、转成 ledger 事件、删源文件。这种解耦让任何能写文件的系统都能集成（包括非 GitHub 的 CI）。

#### 5.6.3 Actor 分级与衰减

不同 actor 产生的 `verified` 事件**权重不同、衰减也不同**：

| actor_id pattern          | verification_boost | 重置 last_event_at | 备注                |
| ------------------------- | ------------------ | ----------------- | ----------------- |
| `human:<email>`           | +0.05              | 是                 | 最强信号              |
| `ci:github-actions`       | +0.03              | 是（短重置：仅 14 天）     | 多次 CI 通过收益递减      |
| `ci:other`                | +0.02              | 是（短重置：14 天）       |                   |
| `agent:claude-code`       | +0.02              | 是（短重置：7 天）        | 必须有外部副作用证明        |
| `agent:other`             | +0.01              | 是（短重置：7 天）        |                   |
| `runbook-exec`            | +0.04              | 是                 | 因为有真实 side effect |
| `test:integration`        | +0.04              | 是                 |                   |
| `test:unit`               | +0.02              | 是（短重置：14 天）       | 单测能验证的范围有限        |

**"短重置"含义**：连续 N 天内的同 actor type 重复 verify 只重置一次。避免 CI 每小时都跑、ledger 被刷屏。

#### 5.6.4 关键边界

明确**不能**自动从这些行为产生 verified 事件：

* **LLM 读完 page 自己说"看起来对"** → 拒绝，循环论证
* **page 被 query 命中** → 拒绝，被搜到不等于内容正确
* **page 被 link** → 拒绝，被引用不等于被验证
* **测试 pass 但没显式 link 到 page** → 拒绝，没有可解释的关联
* **CI 通过但 changed_files 跟 page 无关** → 拒绝

核心原则：**verified 事件必须能 trace 回一个"page 之外的世界发生了什么"的具体证据**。LLM 自言自语不算。

#### 5.6.5 反验证（runtime contradiction）

对称地，runtime 失败要产生反向信号：

* `runbook exec` 失败 → 写 `contradicted_by` 事件，severity 看哪一步失败：早期步骤 major，最后一步 minor
* CI 失败且改了 page 引用的代码 → 写 `contradicted_by` minor（可能是代码错不是文档错，所以是 minor）
* test 标了 `@akb-page X` 但测试失败 → 写 `contradicted_by` major

这是让 confidence 真正反映现实的关键。一个 runbook 一年没人跑过 → 衰减；一个 runbook 上周跑 fail → 直接掉到 0.3 以下。

#### 5.6.6 工程实现位置

`akb runbook` 和 `akb test --link-pages` 实现起来不轻，**整体放进 v0.1.5**。v0.1 只做 runtime signal ingestion：`akb webhook` 和 `akb watch` 接收外部系统已经产生的信号，并把它们转成 ledger 事件。真正执行 runbook / 解析测试标注的强验证路径放到 v0.1.5。

#### 5.6.7 actor_id 必填，禁止空声明

写进代码里的硬约束：**任何 `verified` 或 `contradicted_by` 事件必须有非空 actor_id，且必须能映射到一个可识别的真实主体**。`actor: "agent"` 不带 actor_id 直接 reject。这是防止"LLM 给自己 verify"渗透进系统的最后一道门。

## 6. Patch workflow 集成

v0.1 引入 patch workflow，confidence ledger 在 patch 的几个关键决策点起作用：

### 6.1 Patch 生成时

LLM 看到新 source 矛盾旧 page，必须在 patch metadata 里声明：

```yaml
# .akb/patches/patch_042.yaml
id: patch_042
title: "Update GC threshold per FAST 2026 paper"
sourceIds: [src_007]
changes:
  - type: modify
    pageId: page_gc001
    confidenceImpact:
      kind: contradicted_by
      severity: major
  # OR
  - type: supersede
    oldPageId: page_gc001
    newPageId: page_gc002
status: proposed
```

### 6.2 Patch review 时

UI / CLI 展示当前 page 的 confidence 历史，让 reviewer 知道这页"以前多硬"：

```
$ akb patch show patch_042

Patch: Update GC threshold per FAST 2026 paper
  Source: src_007 (weight 0.5, type=pdf, vendor whitepaper)

  Target: page_gc001 (current confidence 0.71)
    - 3 supporting sources
    - Last verified 102 days ago
    - 0 prior contradictions

  Proposed change: contradicted_by (severity: major)
    Estimated new confidence: 0.36

  Reviewer note: Source weight is low (0.5). Consider gathering corroborating
  source before supersede. Recommended: minor contradiction + needs_verification.
```

### 6.3 Patch apply 时

apply 行为本身是 supersession 决策的物化。完成后写对应事件到 ledger，update `confidence_state`。

## 7. 与 v0.0 的迁移路径

v0.0 没有 ledger。从 v0.0 vault 升级到 v0.1 时：

```bash
akb migrate to-v0.1
```

行为：
1. 为每个 page 创建 `.<id>.ledger.jsonl`
2. 用 frontmatter 里的 `source_path` / `source_hash` 合成稳定 `sourceId`；没有时生成 `src_unknown_<page_id>`
3. 用 frontmatter 里的 `imported_at` 作 source_added 事件时间；没有时 fallback 到 `created_at`
4. 用 frontmatter 里的 `last_verified_at`（如有）作 verified 事件时间
5. 没有 source 信息的 page → 单条 `source_added` with sourceWeight=0.5（"unknown origin"）
6. 跑一次 `akb decay --run` 把当前时间点的衰减算出来
7. 生成 `migration-report.md` 列出哪些 page 现在 score 偏低

这一步是幂等的；可以多次跑。

## 8. 几个故意不做的设计

明确告诉自己**不要**做这些，否则一年内会变成另一个 Notion：

1. **不做"用户分级"**：confidence 不区分 author identity。一个 page 的可信度由 source 和 verification 决定，不是写它的人是谁。
2. **不做 ML 学的 confidence**：所有公式都是确定的、可手算的。不允许用 LLM 给 page 打 confidence 分——那样无法 audit。
3. **不做 claim-level confidence**：粒度细到 claim（LLM 抽出的句子级断言）的方案在工程文档场景反复失败——LLM 抽取准确率 70% 长尾差、reviewer 疲劳、graph-vs-markdown 双向同步是分布式系统级难题。v0.2 会引入 section-level（按 markdown header 切分）confidence 作为中间方案——粒度足够细、边界由 syntax 给定、不依赖 LLM 抽取。
4. **不做 cross-vault propagation**：同一个 page id 在不同 vault 里完全独立。共享知识用 git submodule + supersede 显式 fork，不用神奇的同步。
5. **不让 LLM 自己触发 verify**：`verified` 必须由人类或外部 CI 触发。LLM 给自己写的东西打"已验证"是循环论证。例外：`--by-agent` 模式必须带具体 actor_id 和外部副作用（"agent ran the runbook and it succeeded"），不能空声明。详见 5.6 runtime verification。
6. **不把 confidence 用乘法进 retrieval**：见 3.7，乘法会让 confidence=0.3 的 page 接近隐藏，违背 ledger "标记腐烂、不偷偷藏"的哲学。

## 10. 验证标准

实施 confidence ledger 之后，必须能演示以下场景：

1. **腐烂检测**：用可注入测试时钟把当前时间推进 200 天，跑 `akb lint`，所有 runbook 都被标记 NEEDS_REVIEW
2. **矛盾标记**：ingest 两份相互矛盾的 source，两个相关 page 都 contradiction_count ≥ 1
3. **取代链**：建立三层 supersede 链，默认检索只命中链尾，`--include-superseded` 能拿到全链
4. **可回放**：删除某个 page 的最后 5 个 ledger 事件，跑 `akb confidence recompute`，score 和事件删除前的更早状态完全一致
5. **CI gate**：故意把一个 ADR 标记为 unverified 100 天，PR pipeline fail
6. **可解释**：随便挑一页跑 `akb confidence show`，每一个小数点后的分数都能对到一条事件
7. **Confidence-aware ranking 不藏 page**：构造一个 confidence=0.3 但 BM25 高度匹配的 page，查询时它仍在 top-5 返回，且 flags 含 `NEEDS_REVIEW`。验证 ranker 没有把它隐性删除（这是设计目标的硬底线）。
8. **Runtime signal ingestion 闭环**：构造一个 `.akb/runtime-signals/<timestamp>.json` 或调用 `akb webhook ci-success`，匹配到 page 后写入 `verified` 事件；无法映射到 page、缺 `actor_id`、或证据为空的信号必须被 reject。
9. **Projection rebuild 不丢数据**：删掉 `.akb/index.db`，跑 `akb projection rebuild --all`，所有 confidence_state 与删除前完全一致（用 SQLite dump diff 验证）。
10. **Actor 强制约束**：尝试写一个 `verified` 事件 actor=`agent` 但不带 actor_id，必须被 reject 且返回明确错误。

任何一条做不到，说明 ledger 设计有漏洞，先修再继续。

## 11. 一句话总结

**confidence ledger 不是分数，是知识库的 git log**。Page 的 markdown 文件是"代码"（canonical），frontmatter 里的 confidence 是"运行时状态"（投影），`.ledger.jsonl` 是"事件流"（canonical，入 git），SQLite `confidence_events / confidence_state` 是"查询索引"（投影，不入 git）。四者职责分明、各自可独立演化、共同保证一个长期可信的知识库。

外围三件套（**ranker** 让 confidence 进检索 score 但不藏 page、**runtime verification** 让真实系统行为产生 verified 事件、**section-level confidence** 在 v0.2 解决粒度问题）补完整套机制。

v0.0 把"管道接通"；v0.1 这块把"知识不腐烂"。两块加起来才是一个能用三年的系统底座。
