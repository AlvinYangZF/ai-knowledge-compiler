# AI-Native Knowledge Compiler — v0.0 极简 Spec

> 版本：v0.0
> 日期：2026-05-13
> 目标：用 2-3 周时间把 **ingest → index → MCP → Claude Code 调用** 这个完整 loop 跑通，作为 v0.1 之后所有功能的事实底座。

---

## 0. 这一版的存在理由

v0.1 计划试图一次性交付 10 项 P0 功能（Git vault、CLI、ingest、BM25、vector、MCP、ask、patch、code scan、context pack），实际工作量是 6-9 个月。v0.0 把这个砍到一个**可在 2-3 周内跑通的真实闭环**，剩余功能按 v0.1 / v0.2 增量交付。

v0.0 的核心论点是：**在加任何 LLM synthesis / patch / graph / code intel 之前，先证明"markdown 文件 → 索引 → MCP → Claude Code 调用并引用"这条管道是可靠的**。这条管道跑不通，上面所有花活都是空中楼阁。

v0.0 跟 mcpvault 这种现成方案的差异化只保留两条：**(a) 检索结果带可定位的 citation（页面 ID + 行号范围）**；**(b) 自带 eval 框架，每个 PR 自动跑 golden set 回归**。剩下的差异化（LLM compile、patch、code intel）全部推迟。

## 0.5 这个产品到底是什么——以及为什么不直接用现成轮子

定位一句话：**akb 是 coding agent 的可审计持久记忆层（the persistent, auditable memory layer for coding agents）**。"可审计"是核心差异化关键词——人能读、能改、能 PR review、能用 git blame。这一条 Devin / Cursor / Copilot 的内部 memory 都给不了。

它**不是**：

* 不是另一个 Notion / Confluence（那些不为 agent 设计，也不可 git）
* 不是另一个 Obsidian（那个是给人用的，agent 接口靠插件凑）
* 不是另一个 RAG framework（Dify / RAGFlow / FastGPT 是 query-time 检索，akb 是 ingest-time 编译——v0.1 起的差异）
* 不是 agent 的黑盒 cache（Devin memory / Cursor memory 那种）

它**是**：

* 一个 git-backed markdown vault，**人和 agent 共享同一份事实**
* 一个 MCP-first 接口，让 Claude Code / Codex / Cursor 都能查同一个知识库
* 一个有 confidence 演化机制的"知识 git"——不仅记录"现在认为什么"，也记录"过去认为过什么、为什么改了"（v0.1）

### 0.5.1 为什么不直接用 X

| 方案                                            | 为什么不够                                                                              |
| --------------------------------------------- | ---------------------------------------------------------------------------------- |
| **Obsidian + mcpvault**                       | 缺 citation 行号定位、缺 eval framework、缺 patch workflow、缺 confidence ledger、缺 code intel  |
| **Sourcegraph Cody Context**                  | 重代码图谱，弱文档；闭源 SaaS 锁定；不能本地 git；不可读 markdown                                          |
| **Glean / Notion AI**                         | 不可 git、不可 diff、不为 agent 设计；企业产品定位与单人/小团队不匹配                                         |
| **Dify / RAGFlow / FastGPT / AnythingLLM**    | RAG 引擎导向，query-time 检索；不积累、不演化、没有 patch / supersede 概念                              |
| **Karpathy LLM Wiki（gist 原版）**                | 是好模式但没产品；缺 MCP 接口、缺 citation、缺 code intel、缺工程化的 confidence 计算规则                     |
| **llmwiki / SwarmVault / Pratiyush llm-wiki** | Karpathy 模式的实现尝试，但都偏 single-user demo，没有 agent-native 协议（MCP）、没有评测框架、没有 git-native commit 流 |
| **Devin / Cursor 内部 memory**                  | 黑盒、不可读、agent-internal；你看不见、改不动、不能 PR；锁在 vendor 里                                    |
| **GitHub Copilot Workspace**                  | spec-driven 但无持久记忆层；每个 session 重新理解 repo                                            |
| **Cognee / Graphiti**                         | knowledge graph 系统，方向相近但 graph-native 不是 markdown-native——人难以直接编辑；好的补充，不是替代          |

### 0.5.2 真正的对标和真正的竞争

中期对标：**Karpathy LLM Wiki 模式的工程化实现**（同时具备 MCP 协议 + git 原生 + citation + confidence + code intel）。这是目前还没人把所有零件拼齐的位置。

长期竞争：**coding agent 的工程记忆层**。但**不是要做更好的 Devin memory**，而是做相反方向——human-readable + git-versioned + multi-agent-shared。两者长期会并存（agent 自己的快记忆 + 团队的慢记忆），akb 占的是后者。

### 0.5.3 v0.0 的差异化压缩到最小

为了在 2-3 周内交付，v0.0 只保留两条差异化：**citation 带行号** + **eval 框架**。剩下的差异化（patch workflow、confidence ledger、LLM compile、code intel）全部进 v0.1+。即使只交付 v0.0，跟 mcpvault 比也已经有差，跟 RAG framework 比有"git-native"，跟 Obsidian-only 比有"agent-first MCP"。

## 0.6 数据模型的层次：markdown 是 canonical，其它都是投影

这一节是回应"知识本体是 claim graph 还是 markdown 文档"的一类常见误解。

**核心立场**：**markdown 是事实源（canonical truth），entities / relations / confidence_events / search index 都是从 markdown 派生的投影层（projection layer）**，永远不反过来。

### 0.6.1 为什么不做 claim-first / graph-first

Claim-first / graph-first 架构（让外部 graph 是真相、markdown 是 render layer）在工程文档场景失败过很多次，失败模式高度一致：

* **LLM 抽 claim 准确率长尾差**——70% 准，30% 在边界 case 瞎切。整个系统建在不可靠抽取步骤上，下游每个操作都要处理"抽错了怎么办"
* **人类无法直接编辑**——真相在 graph，markdown 是渲染产物；用户手改 markdown，下次 regenerate 就被盖掉，这是 Confluence-with-AI 系列产品死亡曲线的成因
* **叙事被切碎**——设计文档真正值钱的是"为什么"的推理链、跨段落的因果、作者的取舍踌躇；claim graph 装不下，强行装下也是把生肉切成丁
* **双向写入不一致**——人编辑 markdown、graph 自动更新；graph 改了、需要回写 markdown；中间任一步骤崩了就 split-brain

Sourcegraph / Glean 这条路在代码搜索 work，因为代码本身就高度结构化。在设计文档 / 架构推演 / 取舍记录这种"叙事密集"场景反复失败。

### 0.6.2 投影层的不变量

```
markdown (canonical)
   ↓ derive
projection layer:
   - search index (FTS5, v0.0)
   - vector index (v0.1)
   - entity index (v0.1)
   - relation graph (v0.1)
   - confidence_events table (v0.1)
   - confidence_state cache (v0.1)
```

**关键不变量**：

1. **任何投影任何时候都可以从 markdown 完全重建**——`akb index --rebuild` 是 v0.0 P0；v0.1 起加 `akb projection rebuild --all`
2. **任何投影与 markdown 冲突时，markdown 赢**——投影是缓存，不是 source of truth
3. **任何写操作必须先写 markdown，再更新投影**——不允许直接写投影绕过 markdown
4. **投影层不入 git**——v0.0 已经把 `.akb/index.db` 加进 `.gitignore`；未来所有 projection table 同理

### 0.6.3 这给了什么

* **人可以放心手改 markdown**——下次 reindex 投影会跟上，不会有 ghost 数据残留
* **多 agent / 多机器协作**只需要 git 同步 markdown，投影各自重建
* **灾难恢复就是 `git pull && akb index --rebuild`**——投影 corrupted 也不会丢任何事实
* **审计永远可能**——任何"agent 现在认为 X"的状态都能 trace 回某个 markdown 文件的某个 commit

### 0.6.4 不变量的代价

代价是性能：每次 ingest 都要更新多个投影。v0.0 阶段投影只有一个（FTS5），代价可忽略。v0.1 起会变重，但 incremental update（按 page hash 比对）能把 90% 操作压到毫秒级。这是后续优化空间，不是设计妥协方向。

## 1. 一句话目标

```
用户在一个项目目录跑 akb init + akb ingest + akb mcp serve 后，
Claude Code 能通过 MCP 检索这个知识库并在回答中引用具体页面，
且整个流程在 git 中有完整可追溯历史。
```

## 2. 范围

### 2.1 In scope

* `akb init` 创建 git-backed markdown vault
* `akb ingest` 把 markdown 文件（含 frontmatter）导入 vault
* `akb index` 建立 SQLite FTS5 全文索引
* `akb search` 命令行检索
* `akb mcp serve` 启动 MCP server，暴露 2 个工具
* `akb eval` 跑 golden eval set
* Obsidian 可直接打开生成的 vault

### 2.2 Out of scope（明确推迟）

| 功能            | 推迟到 | 原因                                    |
| ------------- | --- | ------------------------------------- |
| Vector search | v0.1 | BM25 已经能撑前 100 页规模；vector 引入 embed 成本和 LanceDB 依赖 |
| LLM ask       | v0.1 | 先证明 retrieval 准确率，再加生成层                |
| Patch 工作流     | v0.2 | 没有 LLM 写入就没有 patch 需求                  |
| Code intel    | v0.2 | tree-sitter + LSP 是另一个独立子项目            |
| Graph / GraphRAG | v0.3 | 实体抽取需要 LLM，先稳基础                       |
| Web UI        | v0.3 | CLI + MCP 足够验证 loop                    |
| LLM compile（ingest 时合并）  | v0.1 | 这是真正的差异化，但需要先有稳定的索引层 |
| Python worker | v0.1+ | TS 单一运行时足够 v0.0                       |
| `.akbignore`  | v0.1 | v0.0 只接受显式传入的文件                       |

### 2.3 暂不支持的输入类型

v0.0 **只支持 `.md` 文件**。PDF、URL、code repo、issue、PR、chat log 全部 v0.1+。

> v0.1 ingest normalization 已扩展该限制：`akb ingest` 可以把 PDF、Word、文本、markup 文档和显式选择的代码文件规范化为 canonical Markdown 页面。v0.0 本节保留为历史范围说明。

```
┌─────────────────┐
│ 用户 markdown    │

         │ akb index

┌─────────────────┐         ┌──────────────────┐

         │

```

通过标准：用户提一个问题，Claude Code 通过 MCP 检索，**给出的回答里 ≥1 条 citation 链接回 vault 里真实存在且实际相关的页面**。

## 4. 功能详细规约

### 4.1 `akb init <name>`

创建一个 git-backed vault。

```bash
akb init my-knowledge-base
```

执行效果：

```
my-knowledge-base/
├── .git/                         # 自动 git init
├── .akb/
│   ├── config.yaml
│   └── index.db                  # 空 SQLite，schema 已建
├── .gitignore                    # 默认忽略 .akb/index.db
├── README.md                     # 默认欢迎页
└── pages/                        # 所有 markdown 落在这里
    └── .gitkeep
```

`.akb/config.yaml` v0.0 内容：

```yaml
version: "0.0"
workspace:
  name: "my-knowledge-base"
  vault_dir: "."
index:
  engine: "sqlite-fts5"
  path: ".akb/index.db"
mcp:
  host: "127.0.0.1"
  port: 8765
```

自动执行：

* `git init`
* `git add .`
* `git commit -m "akb: initialize vault"`

**Acceptance：** 跑完后 `git log --oneline` 显示 1 条 commit，目录结构如上。

---

### 4.2 `akb ingest <path>`

把一个 markdown 文件或目录（递归）导入 vault。

```bash
akb ingest ./notes/design.md
akb ingest ./old-vault/                 # 递归导入目录
akb ingest ./notes/design.md --tag storage --tag draft
```

行为：

1. 复制目标文件到 `pages/`（保持相对路径结构）
2. 如果原文件没有 frontmatter，**自动补 frontmatter**：
   ```yaml
   ---
   id: page_<random-12-char>
   title: <从首个 # 标题推导，或文件名>
   created_at: 2026-05-13
   updated_at: 2026-05-13
   imported_at: 2026-05-13T10:30:00Z
   source_path: ./notes/design.md
   source_hash: sha256:<original-file-hash>
   ---
   ```
3. 如果原文件已有 frontmatter，**保留并补全缺失字段**（不覆盖已有值）
4. 所有 citation 行号都以最终落在 `pages/` 的 markdown 文件为准，**包含 frontmatter 行**，不是 stripped body 的相对行号
5. 解析 `[[wiki link]]` 但不主动创建空白页（v0.1 再做）
6. `git add` + `git commit -m "akb: ingest <filename>"`
7. 触发增量 index update（见 4.3）

错误处理：

* 文件不是 `.md`：跳过并 warn
* 目标 vault 已存在同 `id`：报错并要求 `--force` 或 `--update`
* 文件含非 UTF-8 字符：跳过并 error log

**Acceptance：** 导入 10 个 markdown 文件后，`pages/` 下有 10 个文件，每个文件 frontmatter 完整，`git log` 有 10 条 commit（或 1 条批量 commit，可配置）。

### 4.3 `akb index [--rebuild]`

构建 / 重建 SQLite FTS5 索引。

```bash
akb index               # 增量
akb index --rebuild     # 全量重建
```

SQLite schema（v0.0）：

```sql
CREATE TABLE pages (
    id              TEXT PRIMARY KEY,
    path            TEXT NOT NULL UNIQUE,
    title           TEXT NOT NULL,
    frontmatter     TEXT NOT NULL,       -- JSON
    content_hash    TEXT NOT NULL,       -- SHA-256 of indexed inputs: frontmatter + body + body_start_line
    body_start_line INTEGER NOT NULL,    -- first body line in the physical file
    indexed_at      TEXT NOT NULL
);

CREATE INDEX idx_pages_path ON pages(path);

CREATE VIRTUAL TABLE pages_fts USING fts5(
    id UNINDEXED,
    title,
    body,
    tags,
    tokenize='unicode61 remove_diacritics 2'
);

CREATE TABLE chunks (
    id              TEXT PRIMARY KEY,
    page_id         TEXT NOT NULL,
    idx             INTEGER NOT NULL,    -- 0-based chunk order within page
    line_start      INTEGER NOT NULL,    -- physical file line, including frontmatter
    line_end        INTEGER NOT NULL,
    text            TEXT NOT NULL,
    token_count     INTEGER NOT NULL,
    FOREIGN KEY (page_id) REFERENCES pages(id) ON DELETE CASCADE
);

CREATE INDEX idx_chunks_page ON chunks(page_id);

CREATE VIRTUAL TABLE chunks_fts USING fts5(
    id UNINDEXED,
    page_id UNINDEXED,
    text,
    tokenize='unicode61 remove_diacritics 2'
);
```

Chunking 策略（v0.0 简单粗暴）：

* 按 markdown header 切分（`#` `##` `###`）
* 每个 chunk 最长 800 token；超长按段落硬切
* **记录每个 chunk 的 `line_start` 和 `line_end`**——这是 citation 的基础；行号必须是 `pages/*.md` 物理文件行号，包含 frontmatter 偏移

增量逻辑：

* 对比 `content_hash`，只重 index 变了的页面；hash 必须覆盖 frontmatter、body 和 `body_start_line`，否则 frontmatter 行数变化会导致 citation offset 过期
* 删除文件时清掉对应记录

**Acceptance：** 10 个文件、约 30 个 chunk 的 vault，全量 index 应在 < 2 秒完成；增量 index 一个修改的文件应在 < 200ms 完成。

### 4.4 `akb search <query>`

命令行 BM25 检索，输出带 citation。

```bash
akb search "garbage collection"
akb search "GC" --top-k 5
akb search "FTL" --format json
```

默认输出（人类可读）：

```
[1] page_abc123  pages/storage/gc.md  L42-L58   score=8.32
    # Garbage Collection Strategy
    > ... we adopt a hybrid GC approach where greedy and FIFO ...

[2] page_def456  pages/storage/ftl.md  L120-L141  score=6.87
    # FTL Internals
    > ... GC is triggered when free block count drops below ...

3 results in 12ms.
```

JSON 输出（agent 友好）：

```json
{
  "query": "garbage collection",
  "results": [
    {
      "page_id": "page_abc123",
      "path": "pages/storage/gc.md",
      "title": "Garbage Collection Strategy",
      "score": 8.32,
      "snippet": "we adopt a hybrid GC approach where greedy and FIFO ...",
      "citation": {
        "line_start": 42,
        "line_end": 58
      }
    }
  ],
  "elapsed_ms": 12
}
```

**Acceptance：** 对 50 页的测试 vault，95% 的查询应在 < 50ms 返回；每个结果必须有可定位到真实文件 line 范围的 citation。测试集必须包含带 frontmatter 的页面，验证 `line_start` / `line_end` 能直接定位到 `pages/*.md`。

### 4.5 `akb mcp serve`

启动 MCP server（stdio + 可选 HTTP）。

```bash
akb mcp serve                     # 默认 stdio
akb mcp serve --transport http    # HTTP on :8765
```

**只暴露 2 个工具**（v0.0 严格控制）：

#### Tool 1: `search_knowledge`

```json
{
  "name": "search_knowledge",
  "description": "Search the local knowledge vault using BM25. Returns page snippets with citation info (page_id + line range).",
  "inputSchema": {
    "type": "object",
    "properties": {
      "query": { "type": "string", "description": "Search query in natural language or keywords." },
      "top_k": { "type": "integer", "default": 5, "minimum": 1, "maximum": 20 }
    },
    "required": ["query"]
  }
}
```

返回格式与 `akb search --format json` 一致。

#### Tool 2: `get_page`

```json
{
  "name": "get_page",
  "description": "Retrieve the full content of a knowledge page by id or path. Use this after search_knowledge when you need more context than the snippet.",
  "inputSchema": {
    "type": "object",
    "properties": {
      "page_id_or_path": { "type": "string" }
    },
    "required": ["page_id_or_path"]
  }
}
```

返回：

```json
{
  "page_id": "page_abc123",
  "path": "pages/storage/gc.md",
  "frontmatter": { /* parsed yaml */ },
  "content": "# Garbage Collection Strategy\n\n...",
  "line_count": 87
}
```

Claude Code 配置示例（要写进 README）：

```json
{
  "mcpServers": {
    "akb": {
      "command": "akb",
      "args": ["mcp", "serve"],
      "cwd": "/path/to/your/vault"
    }
  }
}
```

**Acceptance：** Claude Code 用上面的配置启动，能在对话中调用 `search_knowledge` 和 `get_page`，并在回答里以 `[page_id]` 形式标注引用来源。

### 4.6 `akb eval`

跑 golden eval set，输出 retrieval 指标。

```bash
akb eval
akb eval --set ./my-eval.yaml --output report.json
```

Golden set 格式：

```yaml
# .akb/eval/golden.yaml
version: "1.0"
items:
  - id: q001
    query: "garbage collection strategy"
    must_hit_pages:
      - page_abc123           # 必须出现在 top-5
    should_hit_pages:
      - page_def456           # 出现给加分但不强制
    notes: "core GC design question"

  - id: q002
    query: "how is FTL implemented"
    must_hit_pages:
      - page_def456
```

输出：

```
Eval: 10 items
  precision@5:  0.84
  recall@5:     0.91
  must-hit pass rate:  9/10 (90%)

FAILED:
  q007 "what changed in last release"
    expected: page_xyz789 in top-5, got rank 12
```

退出码：

* 0：所有 must-hit 都满足
* 1：有 must-hit 失败（CI 应该 fail）

**Acceptance：** 在 CI（GitHub Actions）里能跑通；每次 PR 自动跑；指标回归（precision@5 下降 > 5%）就 fail build。

## 5. 数据模型（v0.0 子集）

只保留 **Page**。Entity / Relation / Patch 全部留给 v0.1+；Source 在 v0.0 不作为独立 canonical entity，只作为 Page frontmatter 中的来源元数据存在。

### 5.1 Page（markdown frontmatter）

```yaml
---
id: page_abc123              # 必填，自动生成
title: "GC Strategy"          # 必填
type: note                    # 可选：note | concept | design（v0.0 不强制约束）
tags: ["storage", "gc"]      # 可选
aliases: []                   # 可选
created_at: "2026-05-13"
updated_at: "2026-05-13"
imported_at: "2026-05-13T10:30:00Z"
source_path: "./old/gc.md"   # 可选，原始来源
source_hash: "sha256:<original-file-hash>"  # 可选，原始文件 hash
---
```

### 5.2 Source metadata（Page frontmatter）

v0.0 不建独立 `Source` 表。导入来源写在 Page frontmatter 中：

```yaml
source_path: "./old/gc.md"
source_hash: "sha256:<original-file-hash>"
imported_at: "2026-05-13T10:30:00Z"
```

原因：SQLite 是投影层，不入 git；如果 audit 事实只存在 SQLite，`akb index --rebuild` 后会丢。v0.1 的 compile / patch workflow 如果需要稳定 `SourceId`，必须先引入入 git 的 canonical source log，再从它派生 SQLite 表。

## 11. v0.0 → v0.1 升级触发条件

v0.0 完成后**不要立即扩功能**。先用 4-6 周观察：

1. 自己用：把它接进现有的 storage/infra 笔记，记录哪里痛
2. 让 1-2 个朋友试：观察他们最先要的功能是什么
3. 跑 eval：在真实 vault 上 precision@5 能不能稳定 > 0.7

满足以下任一条件再进 v0.1：

* 真实 vault 规模 > 50 页且 BM25 准确率明显不够（→ 加 vector）
* 用户反复表达"想直接问问题不想看 snippet"（→ 加 ask）
* 多人协作场景出现（→ 加 patch workflow）

如果上面三条都没出现，**说明 v0.0 已经够用**——不要为了 roadmap 而加功能。

## 13. 关键设计纪律

写在最前面，免得被自己 scope creep：

1. **任何要 LLM API key 的功能都不在 v0.0**（除非用户在 Claude Code 那边自己用）。这保证 v0.0 跑通不需要任何外部账号。
2. **任何要 native build 的依赖都需要单独评审**（除 better-sqlite3）。
3. **v0.1 起默认 LLM provider 是 DeepSeek**；DeepSeek、OpenAI、Anthropic 的 provider、model、base URL 和 API key 环境变量名可配置，真实 secret 只放在本机环境变量中。
4. **任何加新 MCP tool 的提议都要先证明现有 2 个不够**。
5. **任何让 LLM 写入 vault 的功能都推迟**——v0.0 vault 的所有写入都来自人或来自 `akb ingest` 的纯文件拷贝。
6. **每个 PR 必须跑通 eval**，precision@5 不允许回归 > 5%。
7. **任何投影层数据（index、vector、graph、ledger 缓存）都不入 git**——这些是 markdown 的派生物，git 只追事实源。
8. **任何"修改 vault"的代码路径都必须先写 markdown 再更新投影**——绕过 markdown 直接写投影是 architectural violation，PR 直接拒。

## 14. 一句话总结

**v0.0 的成就是"有"，不是"好"**：让 ingest → index → MCP → Claude Code 这条管道在一个 git-backed vault 上端到端跑通，并用 eval 锁住底线。所有"compile"、"compile-time synthesis"、"patch"、"graph"、"code intel" 都是 v0.1+ 的故事。

打好这块底，后续每个版本都能在稳定底座上加东西；底座没打稳就上花活，三个月后会变成一个谁也维护不动的半成品。
