# LLM Compile 设计规约（v0.1）

> 配套 v0.0 spec 和 confidence ledger 规约。
> 解决问题：让 ingest 从"纯文件拷贝 + 索引"升级成"理解新 source 并把它编译进现有知识"——这是产品名字里"Compiler"四个字的兑现。
> 状态：v0.1 本地实现已落地，包含 provider-backed compile、heuristic fallback、patch workflow、lineage 和 replay。
> 前置依赖：v0.0 的 ingest / search-engine / markdown-engine；v0.1 的 confidence ledger（§ 全文）、patch workflow；v0.0 spec § 0.6 投影层原则。
> 默认 LLM provider：DeepSeek。当前支持 DeepSeek、OpenAI、Anthropic；`.akb/config.yaml` 只记录 `api_key_env`，真实 API key 留在本机环境变量中，避免误提交 secret。

---

## 0. 为什么这是产品的核心，不是可选项

### 0.1 名字的承诺

产品叫 "Knowledge Compiler"。这个名字隐含一个具体的技术承诺：

> **昂贵的、累积性的知识组织工作发生在 ingest 时（compile time），不是 query 时。**

这跟所有 RAG 系统的根本区别就在这里。RAG 在 query 时才做检索、拼接、组织——每次查询都重新发现一遍知识结构，本质是"无状态重计算"。Compiler 模式在 ingest 时就把新知识**编译进**已有知识体系：更新相关 entity 页、标注与旧 claim 的矛盾、改写受影响的 synthesis 页。query 时只做"检索 + 重排"这种廉价操作。

如果 `akb ingest` 永远只是"拷贝文件 + 建索引"，那这个产品就是个 RAG 引擎，名字应该叫 "Knowledge Index"。`akb compile` 是兑现 "Compiler" 这个词的那个命令。

### 0.2 不做 compile 的退化路径

v0.0 的 ingest 是纯拷贝。如果 v0.1 不补 compile，知识库会这样退化：

* **碎片化**：ingest 10 篇关于 GC 的笔记 → 10 个互不知道对方存在的 page。检索"GC"返回 10 个片段，agent 自己拼。
* **无累积**：第 11 篇笔记进来，不会去更新前 10 篇里过时的部分。知识只增不合。
* **矛盾不可见**：第 11 篇说的跟第 3 篇矛盾，没有任何机制发现——除非有人正好同时检索到这两页。
* **synthesis 缺失**：永远没有一个"GC 当前认知综述"页。每次想要全貌都要 agent 现场重新综合。

confidence ledger 解决的是"知识会腐烂"，compile 解决的是"知识不会自己长在一起"。两个问题正交，都得解决。

### 0.3 与 Karpathy LLM Wiki 模式的关系

这一节本质是 Karpathy LLM Wiki "ingest workflow" 的工程化实现。Karpathy gist 里的核心描述：ingest 新 source 时 LLM 不只是索引，而是**读取、抽取要点、整合进现有 wiki——更新实体页、修订主题摘要、标注新数据与旧主张的矛盾**。

`akb compile` 把这个描述变成：确定的 pipeline、可 review 的 patch、可 audit 的 lineage、可回归的 eval。Karpathy 给了模式，这份 spec 给工程。

## 1. 设计目标与非目标

### 1.1 目标

1. **ingest 一个新 source 时，自动发现它与哪些现有 page 相关**，并生成"如何把它编译进去"的 patch 提议。
2. **所有 compile 产物都是 patch，不是直接写入**——人或 agent review 后才 apply。这是 v0.0 spec § 13 纪律 4 的延续。
3. **compile 产生的每一个 chunk / page 都带 lineage**——能 trace 回它是从哪些 source chunk、用什么方法、什么 prompt 生成的。
4. **compile 是确定性可复现的**——同样的输入 + 同样的 prompt + pinned model = 同样的 patch（在 LLM 采样允许的范围内，用 temperature=0 + prompt hash 锁定）。
5. **compile 失败可降级**——LLM 不可用 / 超时 / 输出不合法时，自动退回 v0.0 的纯拷贝 ingest，不阻塞。Provider-backed compile 的单次 LLM 请求默认 120 秒超时；classify relation 或 synthesize patch changes 不符合 schema 时会带校验错误自动重试一次。

### 1.2 非目标

1. **不做 query-time synthesis**——compile 是 ingest-time。query 时的组织工作只有"检索 + 重排"。
2. **不做自动 apply**——compile 永远只到 "patch proposed" 为止。apply 是 patch workflow 的事，需要 review gate。
3. **不做 claim 抽取**——见 confidence ledger § 8.3，不走 claim-first。compile 的产物是 markdown page 和 markdown section，不是 claim 节点。
4. **不做跨 vault compile**——只在当前 vault 内编译。
5. **不替代人写文档**——compile 是"把新 source 整合进已有结构"，不是"从零生成知识库"。冷启动的 vault 还是要人先写骨架。

## 2. Compile 在数据流里的位置

```
akb ingest <source>
   │
   ├─[v0.0 行为]─ 拷贝文件到 pages/ ─ 建 FTS5 索引 ─ git commit
   │
   └─[v0.1 新增]─ akb compile（可被 ingest 自动触发，也可单独跑）
         │
         ▼
   ┌──────────────────────────────────────────────┐
   │ Compile Pipeline（5 阶段）                      │
   │                                              │
   │  1. Segment    新 source 切成 semantic units   │
   │  2. Locate     找出每个 unit 关联的现有 page    │
   │  3. Classify   判定关系：new / extend /        │
   │                merge / contradict / supersede │
   │  4. Synthesize LLM 生成具体的 patch 内容        │
   │  5. Emit       写出 patch 文件 + lineage       │
   └──────────────────────────────────────────────┘
         │
         ▼
   .akb/patches/patch_NNN.yaml  (status: proposed)
         │
         ▼
   [patch workflow]  akb patch list / show / apply
         │
         ▼
   markdown 被修改 → 投影层更新 → confidence ledger 写事件
```

**关键**：compile 自己**不碰 markdown**。它只产出 patch。markdown 的实际修改发生在 `akb patch apply`，由 patch workflow 负责。这保证了 v0.0 spec § 13 纪律 7（先写 markdown 再更新投影）和 review gate 不被绕过。

## 3. Compile Pipeline 详解

### 3.1 阶段 1：Segment（语义切分）

把新 source 切成 **semantic units**——比 v0.0 的 `chunkByHeaders` 更粗、更语义化的单位。

v0.0 的 chunk 是检索单位（按 header + token 上限切）。compile 的 semantic unit 是**理解单位**：一个 unit 应该承载一个相对完整的论点 / 概念 / 决策。

```typescript
interface SemanticUnit {
  id: string;                    // "su_" + ulid
  sourceId: SourceId;
  sourceChunkIds: string[];      // 对应的 v0.0 chunk（一个 unit 可能跨多个 chunk）
  text: string;
  kind: "concept" | "claim_cluster" | "decision" | "procedure" | "data" | "narrative";
  lineRange: { start: number; end: number };
}
```

切分策略（两段）：

1. **结构预切**：先用 v0.0 的 `chunkByHeaders` 拿到 syntax-level chunk。
2. **LLM 语义合并/拆分**：把结构 chunk 喂给 LLM，让它判断哪些 chunk 应该合成一个 unit（同一论点跨了多个小标题），哪些 chunk 内部其实有多个独立概念需要拆开。

LLM prompt 要点（不是完整 prompt，是约束）：

```
You are segmenting a document into semantic units for knowledge compilation.
A semantic unit carries ONE coherent idea, decision, concept, or procedure.

Rules:
- Output ONLY a JSON array, no prose, no markdown fences.
- Each unit references the input chunk ids it covers.
- Do NOT rewrite or summarize the text — only group/split.
- A unit's text must be a verbatim concatenation of its source chunks
  (you may only choose boundaries, not edit content).
- kind must be one of: concept | claim_cluster | decision | procedure | data | narrative
```

**为什么 segment 阶段不允许 LLM 改写文本**：保持 lineage 干净。segment 只决定边界，不产生新内容。改写发生在阶段 4，且会被标记成 derived。

降级：LLM 不可用时，semantic unit = 结构 chunk，1:1 映射。compile 仍能继续，只是粒度粗一点。

### 3.2 阶段 2：Locate（关联定位）

对每个 semantic unit，找出 vault 里**已经存在的、与之相关的 page**。

```typescript
interface LocateResult {
  unitId: string;
  candidates: Array<{
    pageId: PageId;
    relevanceScore: number;      // 0-1
    matchedVia: ("fts" | "title" | "alias" | "tag" | "wikilink")[];
  }>;
}
```

实现：**不用 LLM**，用 v0.0 已有的 search-engine。

1. 用 unit.text 做一次 FTS5 检索 → 拿 top-N 候选 page
2. 用 unit 里的关键名词短语匹配 page 的 title / alias / tag
3. 合并去重，按 relevanceScore 排序

为什么不用 LLM：locate 是召回问题，FTS5 够用且便宜且确定。LLM 留给阶段 3 的判断和阶段 4 的生成。relevanceScore 低于阈值（默认 0.3）的 unit → 走"new page"路径，不进 LLM 关系判定。

### 3.3 阶段 3：Classify（关系判定）

这是第一个**必须用 LLM** 的阶段。对每个 (unit, candidate_page) 对，判定它们的关系。

```typescript
type CompileRelation =
  | "new"          // unit 是全新知识，候选 page 其实不相关 → 建新 page
  | "extend"       // unit 给 page 补充了新内容，不冲突 → 追加 section
  | "merge"        // unit 跟 page 讲同一件事，表述不同 → 合并改写
  | "contradict"   // unit 跟 page 有事实冲突，但不足以取代 → 标 contradiction
  | "supersede"    // unit 代表对同一主题的实质性更新 → 建议 supersede 链
  | "duplicate";   // unit 跟 page 内容实质重复 → 跳过，不产 patch

interface ClassifyResult {
  unitId: string;
  pageId: PageId;
  relation: CompileRelation;
  confidence: number;            // LLM 对这个判定本身的置信度 0-1
  reasoning: string;             // 一句话理由，写进 patch 供 review
  conflictSpans?: Array<{        // 仅 contradict / supersede 时有
    pageLineRange: { start: number; end: number };
    unitExcerpt: string;
    severity: "minor" | "major";
  }>;
}
```

LLM prompt 约束要点：

```
You are classifying the relationship between a NEW knowledge unit and an
EXISTING knowledge page.

You must pick exactly one relation: new | extend | merge | contradict |
supersede | duplicate.

Critical rules:
- "contradict" vs "supersede": contradict = they disagree on a detail but
  both may have partial validity; supersede = the new unit represents a
  fundamental, methodology-level update that makes the old page outdated.
  When unsure, prefer "contradict" (less destructive).
- "merge" vs "extend": merge = same topic, overlapping content, needs
  rewriting to unify; extend = new sub-topic, can be appended cleanly.
- For contradict/supersede you MUST cite the specific conflicting line
  ranges in the existing page.
- Output ONLY JSON. Include your own confidence in the judgment.
- Do NOT propose the actual edit here — only classify.
```

**这一步故意不让 LLM 写编辑内容**——分离"判断"和"生成"。判断错了（比如把 extend 误判成 supersede）在 review 时容易看出来；如果判断和生成混在一步，review 时要同时验证两件事，认知负担翻倍。

`relation` 的下游影响：

| relation     | 产出 patch 类型          | 影响 confidence ledger             |
| ------------ | -------------------- | -------------------------------- |
| `new`        | create page          | 新 page 写 `source_added`          |
| `extend`     | modify page（追加 section） | 目标 page 写 `source_added`         |
| `merge`      | modify page（改写 section） | 目标 page 写 `source_added`         |
| `contradict` | modify page（插入矛盾标注） + 可能 modify 别处 | 目标 page 写 `contradicted_by`      |
| `supersede`  | create page + supersede 链 | 旧 page 写 `superseded_by`，新写 `supersedes` |
| `duplicate`  | 无 patch              | 目标 page 写 `source_added`（来源加强，不改内容） |

注意 `duplicate` 也写 `source_added`：内容虽然重复，但"又一个独立来源说了同样的话"是对现有 page 的 confidence 加强信号（confidence ledger § 3.2 source strength）。

### 3.4 阶段 4：Synthesize（生成 patch 内容）

第二个**必须用 LLM** 的阶段。只对 `extend / merge / contradict / supersede / new` 的判定结果生成实际编辑内容。

每种 relation 用不同的 sub-prompt。核心约束统一如下：

```
You are generating the actual content for a knowledge patch.

Universal rules:
- Output must be valid against the patch schema (you will be given it).
- Every sentence you write that is NOT directly from the source unit must
  be marked — wrap inferred/synthesized sentences so the lineage tracker
  can flag them. Verbatim content from the source needs no marking.
- Preserve the existing page's voice and structure where possible.
- For "merge": produce the unified section, and a short rationale of what
  changed and why.
- For "contradict": do NOT delete the existing claim. Insert a clearly
  marked contradiction note that references both sides.
- For "supersede": write the NEW page. It must open with a link back to
  the superseded page and a one-paragraph summary of what changed.
- Never invent citations, dates, numbers, or names not present in the source.
```

#### 3.4.1 各 relation 的生成产物

**extend** — 生成一个新 section，追加到目标 page 末尾（或指定 anchor 后）：

```yaml
change:
  type: modify
  pageId: page_gc001
  operation: append_section
  anchor: end                       # 或某个 section id
  content: |
    ## Adaptive Threshold (added 2026-05-14)

    <!-- akb:derived source=su_xxx method=extend -->
    Recent measurements suggest the GC trigger threshold should adapt to
    workload write-amplification rather than staying fixed at 10%.
```

**merge** — 生成替换某个现有 section 的内容：

```yaml
change:
  type: modify
  pageId: page_gc001
  operation: replace_section
  targetSection: s_trigger          # 现有 section 的稳定 id
  content: |
    ## Trigger Conditions

    <!-- akb:derived source=su_xxx,page_gc001:s_trigger method=merge -->
    GC is triggered when the free block count drops below a threshold.
    The original design used a fixed 10%; later analysis (src_007) showed
    a workload-adaptive threshold performs better. See contradiction note
    below for the unresolved part.
  rationale: |
    Merged the original fixed-threshold description with src_007's
    adaptive-threshold finding. Kept both; flagged the open question.
```

**contradict** — 不改原内容，插入矛盾标注块：

```yaml
change:
  type: modify
  pageId: page_gc001
  operation: insert_after_section
  targetSection: s_trigger
  content: |
    > [!contradiction] Conflicting source (2026-05-14)
    > <!-- akb:derived source=su_xxx method=contradict severity=minor -->
    > [[src_007]] reports that a 5% threshold outperforms the 10% stated
    > above under write-heavy workloads. Not yet resolved — both retained.
```

**supersede** — 生成全新 page：

```yaml
change:
  type: create
  newPageId: page_gc002
  path: pages/storage/gc-adaptive.md
  supersedes: page_gc001
  content: |
    ---
    id: page_gc002
    title: "Adaptive Garbage Collection Strategy"
    type: design
    supersedes: page_gc001
    created_at: "2026-05-14"
    ---

    > Supersedes [[page_gc001]]. Change: the GC trigger threshold is no
    > longer a fixed 10% but adapts to workload write-amplification.
    > Historical fixed-threshold reasoning preserved in [[page_gc001]].

    # Adaptive Garbage Collection Strategy

    <!-- akb:derived source=su_xxx method=supersede -->
    ...
```

**new** — 跟 supersede 类似但没有 `supersedes` 字段，是一个孤立新 page。

#### 3.4.2 `<!-- akb:derived -->` 标记

这是 lineage 的 markdown 落地形式。任何由 LLM 生成（而非从 source 逐字搬运）的内容块，前面加一行 HTML 注释：

```
<!-- akb:derived source=<逗号分隔的 unit/section id> method=<compile method> [severity=...] -->
```

* HTML 注释 → Obsidian / 渲染时不可见，但人 review markdown 时看得到
* 解析后进 chunk lineage（见 § 4）
* `akb lint` 会统计 derived 内容占比，过高的 page 标记 "heavily synthesized, needs human review"

### 3.5 阶段 5：Emit（产出 patch + lineage）

把前四阶段的结果组装成 patch 文件，写到 `.akb/patches/`。

一个 compile run 通常产生**一个 patch 文件，包含多个 change**（一个新 source 可能同时 extend 一页、contradict 另一页、新建第三页）。

```yaml
# .akb/patches/patch_042.yaml
id: patch_042
title: "Compile: src_007 (FAST 2026 adaptive GC paper)"
createdAt: "2026-05-14T10:30:00Z"
createdBy: "akb-compile"
status: proposed

source:
  sourceId: src_007
  ingestPath: "./papers/fast2026-adaptive-gc.md"

compileMeta:
  provider: "deepseek"
  baseUrl: "https://api.deepseek.com"
  modelId: "deepseek-v4-flash"             # pinned，可复现
  promptHashes:
    segment: "sha256:a1b2..."
    classify: "sha256:c3d4..."
    synthesize: "sha256:e5f6..."
  pipelineVersion: "compile/0.1"
  segmentCount: 4
  llmCallCount: 9
  elapsedMs: 12400
  degraded: false                          # true = LLM 某阶段降级了

changes:
  - type: modify
    pageId: page_gc001
    operation: insert_after_section
    targetSection: s_trigger
    relation: contradict
    classifyConfidence: 0.78
    reasoning: "src_007 disagrees on threshold value but methodology overlaps"
    content: |
      > [!contradiction] ...
    confidenceImpact:
      kind: contradicted_by
      severity: minor

  - type: create
    newPageId: page_gc002
    relation: supersede
    classifyConfidence: 0.62
    reasoning: "adaptive threshold is a methodology-level change"
    supersedes: page_gc001
    content: |
      ---
      ...

lineage:
  units:
    - id: su_001
      sourceChunkIds: ["src_007_chunk_0", "src_007_chunk_1"]
      kind: claim_cluster
  derivedChunks:
    - chunkId: page_gc002:c0
      derivedFrom:
        sourceUnitIds: ["su_001"]
        method: supersede
        promptHash: "sha256:e5f6..."
```

**低置信度处理**：`classifyConfidence < 0.5` 的 change 不会被丢弃，但在 patch 里标 `needsCloseReview: true`，`akb patch show` 会高亮。compile 的判断不确定时，把决定权交回人，而不是自作主张跳过或自作主张执行。

## 4. Chunk Lineage Schema

这是 ChatGPT review 里点出的真问题：v0.0 的 chunk 没有 lineage，一旦 LLM compile 产生二级 chunk（summary、改写、合并），provenance 就断了。

### 4.1 v0.0 chunk 的扩展

v0.0 spec § 4.3 的 `chunks` 表和 search-engine skeleton 的 `Chunk` 接口，在 v0.1 扩展（**v0.0 不实现这些字段，v0.1 compile 落地时一起加**）：

```typescript
interface Chunk {
  // ===== v0.0 已有字段 =====
  id: string;
  pageId: PageId;
  index: number;
  lineStart: number;
  lineEnd: number;
  text: string;
  tokenCount: number;

  // ===== v0.1 新增：lineage =====
  origin: ChunkOrigin;
}

type ChunkOrigin =
  | { kind: "verbatim" }                      // 从 ingest 的 source 逐字而来（v0.0 默认全是这种）
  | { kind: "derived"; derivedFrom: DerivedFrom };

interface DerivedFrom {
  sourceUnitIds: string[];        // 上游 semantic unit
  sourceChunkIds: string[];       // 进一步追到的原始 chunk（可跨 page）
  method: CompileMethod;
  compiledAt: string;             // ISO 8601
  patchId: string;                // 哪个 patch apply 后产生的
  promptHash: string;             // 可复现
  modelId: string;
}

type CompileMethod =
  | "segment"        // 仅重新切分边界（理论上不产 derived，留作完整性）
  | "extend"
  | "merge"
  | "contradict"
  | "supersede"
  | "summary";       // v0.2 的 synthesis page 会用
```

### 4.2 SQLite schema 扩展

```sql
-- v0.1 给 chunks 表加一列
ALTER TABLE chunks ADD COLUMN origin_kind TEXT NOT NULL DEFAULT 'verbatim';

-- lineage 明细单独一张表（一个 derived chunk 可能有多个上游）
CREATE TABLE chunk_lineage (
    id              TEXT PRIMARY KEY,        -- ulid
    chunk_id        TEXT NOT NULL,
    source_unit_id  TEXT,                    -- nullable: 有些上游只到 chunk 级
    source_chunk_id TEXT,
    method          TEXT NOT NULL,
    patch_id        TEXT NOT NULL,
    prompt_hash     TEXT NOT NULL,
    model_id        TEXT NOT NULL,
    compiled_at     TEXT NOT NULL,
    FOREIGN KEY (chunk_id) REFERENCES chunks(id) ON DELETE CASCADE
);

CREATE INDEX idx_lineage_chunk ON chunk_lineage(chunk_id);
CREATE INDEX idx_lineage_source_chunk ON chunk_lineage(source_chunk_id);
```

`chunk_lineage` 是投影层（不入 git）。canonical 的 lineage 信息在两个地方：markdown 里的 `<!-- akb:derived -->` 注释 + patch 文件的 `lineage` 段（patch 文件 apply 后归档进 `.akb/patches/applied/`，入 git）。SQLite 表是从这两者重建的。

### 4.3 Lineage 能回答的问题

加了 lineage 之后，以下查询变得可能（都通过 `akb lineage` 命令，§ 5.3）：

* **正向**："这个 chunk 是怎么来的？" → 一条 derived 链，最终落到某些 verbatim source chunk
* **反向**："这个 source 影响了哪些 page？" → 从 source_chunk_id 反查所有 derived chunk
* **污染检测**："如果 src_007 被证明是错的，哪些 page 需要重审？" → 反向遍历 lineage 图
* **derived 占比**："这个 page 有多少内容是 LLM 合成的 vs 逐字搬运的？" → 统计 origin_kind
* **可复现性**："重新跑这个 compile 能得到一样的结果吗？" → 用 promptHash + modelId + source 重放

### 4.4 Lineage 断裂的防护

一个 derived chunk 的所有上游必须存在。两个保护：

1. **apply 时校验**：`akb patch apply` 在写入 derived chunk 前，校验它声明的所有 `sourceUnitIds / sourceChunkIds` 都能解析到。解析不到 → 拒绝 apply，报错。
2. **删除时级联标记**：如果一个 source page 被删除，所有 `derivedFrom` 指向它的 chunk 不会被自动删（那样会丢知识），但会被 `akb lint` 标记为 `orphaned_lineage` —— "这块内容的来源已经不在了，需要人确认是否还可信"。

## 5. CLI 命令

### 5.1 `akb compile`

```bash
# 对一个还没 compile 的 source 跑编译
akb compile --source src_007

# ingest 时自动触发（默认行为，可关）
akb ingest ./papers/fast2026.md --compile
akb ingest ./papers/fast2026.md --no-compile     # 退回 v0.0 纯拷贝

# 对整个 vault 里所有 "已 ingest 未 compile" 的 source 批量跑

# 只跑前两阶段，看看会关联到哪些 page，不调用生成
akb compile --source src_007 --dry-run

# 锁定 model
akb compile --source src_007 --model deepseek-v4-pro
```

```bash

```

只需要设置实际使用的 provider 对应的 key。环境变量缺失时降级到 heuristic compile，并在 `compileMeta.degraded=true` 中记录原因。

行为：
1. 跑 5 阶段 pipeline
2. 产出 `.akb/patches/patch_NNN.yaml`，status=proposed
3. **不修改任何 markdown**
4. 打印摘要：

```

  4 semantic units
  changes proposed:
    - modify page_gc001    (contradict, confidence 0.78)
    - create page_gc002    (supersede, confidence 0.62)  ⚠ needs close review
  2 changes, 9 LLM calls, 12.4s
  Review with: akb patch show patch_042
```

降级行为：任一 LLM 阶段失败 → 该 source 退回纯拷贝 ingest，`compileMeta.degraded=true`，打印 warning，**不报错退出**（exit 0）。compile 是增强，不是阻塞。classify 阶段如果返回了不支持的 relation 或非 JSON object，synthesize 阶段如果返回了不合法 patch changes，都会先把校验错误反馈给模型重试一次；重试仍失败才降级。

## 6. 与 confidence ledger 的集成

compile 是 confidence ledger 事件的主要生产者之一。对应关系在 § 3.3 的表里给过，这里说清楚**时序**：

```
akb compile         → 只产 patch，不写任何 ledger 事件
akb patch apply     → 修改 markdown + 写 ledger 事件 + 更新投影
```

ledger 事件**只在 patch apply 时写**，不在 compile 时写。理由：compile 的产物是"提议"，提议被拒绝就不该在 ledger 留痕。ledger 记录的是"实际发生的知识演化"，不是"被考虑过的演化"。

被拒绝的 patch 归档到 `.akb/patches/rejected/`（入 git，留作审计），但不产生任何 ledger 事件。

一个例外：**`duplicate` 关系**。compile 判定某 unit 与现有 page 重复，不产 change，但这本身是一个"又一个来源印证了这页"的信号。这种情况 compile 会产一个**只含 confidence 影响、不含内容修改的 minimal change**：

```yaml
changes:
  - type: confidence_only
    pageId: page_gc001
    relation: duplicate
    confidenceImpact:
      kind: source_added
      sourceWeight: 0.8
```

apply 这个 change 时只写 ledger 事件，不碰 markdown。

## 7. Compile-time vs Query-time 的工作划分

这张表是产品叫 "Compiler" 的技术兑现，明确什么工作在哪一侧：

| 工作                          | 何时做          | 谁做                    |
| --------------------------- | ------------ | --------------------- |
| 切分 source 成语义单位             | compile time | `akb compile` 阶段 1    |
| 发现新知识与旧知识的关联                | compile time | `akb compile` 阶段 2    |
| 判定 new/extend/merge/contradict/supersede | compile time | `akb compile` 阶段 3（LLM） |
| 生成合并后的 section / 新 page      | compile time | `akb compile` 阶段 4（LLM） |
| 标注矛盾                        | compile time | `akb compile` 阶段 4    |
| 建立 supersede 链              | compile time（提议）/ apply time（落地） | compile + patch apply |
| 写 confidence ledger 事件      | apply time   | `akb patch apply`     |
| 更新 FTS5 / vector 索引         | apply time   | patch apply 触发投影更新    |
| ───────────────────────     | ───          | ───                   |
| 检索相关 chunk                  | query time   | search-engine（BM25）   |
| 重排（confidence-aware）        | query time   | ranker（confidence ledger § 3.7） |
| 拼 snippet                   | query time   | search-engine         |
| 生成回答                        | query time   | `akb ask`（v0.1）/ 外部 agent |

**原则**：query time 只剩"检索 + 重排 + 取内容"——全是廉价的、无状态的、确定的操作。所有"昂贵的、累积的、需要 LLM 判断的"工作都被推到 compile time，且只做一次。

这正是与 RAG 的本质区别：RAG 把组织工作放 query time 每次重做；Compiler 放 compile time 做一次、固化进 markdown。

---

## 8. 几个故意不做的设计

1. **compile 不自动 apply**——永远停在 proposed。哪怕 classifyConfidence=0.99 也要过 review gate。自动 apply 是把"知识库正确性"赌在 LLM 单次判断上，不可接受。
2. **compile 不删内容**——contradict 是插入标注，supersede 是建新页 + 链接，merge 是改写但保留信息。没有任何 compile path 会净删除知识。删除只能由人通过显式 patch 做。
3. **compile 不抽 claim**——产物是 markdown section / page，不是 claim 节点。理由见 confidence ledger § 8.3。
4. **compile 不做 query-time synthesis**——`akb ask` 不会触发 compile。ask 只检索已 compile 好的知识。
5. **compile 不跨 source 联合推理**——一次 compile 只处理一个新 source 对现有 vault 的影响。"把 5 篇论文综合成一篇综述"是 v0.2 的 synthesis page 功能，不是 compile。
6. **compile 不改 prompt 不留痕**——promptHash 进 patch。换了 prompt 就是换了 pipelineVersion，老 patch 的可复现性靠归档的 hash 保证。
7. **degraded 不静默**——LLM 降级一定打 warning + 在 compileMeta 标记。绝不假装 compile 成功了。

## 9. 实现工作量估算（v0.1）

| 组件                                      | 估算    |
| --------------------------------------- | ----- |
| `packages/compile` 新 package 骨架          | 1 天   |
| 阶段 1 Segment（结构预切复用 v0.0 + LLM 语义合并）   | 2 天   |
| 阶段 2 Locate（复用 search-engine）           | 1 天   |
| 阶段 3 Classify（LLM + schema 校验 + 重试）     | 2.5 天 |
| 阶段 4 Synthesize（5 个 sub-prompt + 校验）    | 3.5 天 |
| 阶段 5 Emit（patch 组装 + lineage 写出）        | 1.5 天 |
| Chunk lineage schema + SQLite 迁移        | 1 天   |
| `<!-- akb:derived -->` 解析器 + 写入器        | 1 天   |
| `akb compile` 命令 + 降级逻辑                 | 1.5 天 |
| `akb compile status`                    | 0.5 天 |
| `akb lineage`（正向 + 反向）                  | 1.5 天 |
| `akb compile replay`                    | 1 天   |
| 与 patch workflow 集成（confidence_only change 等） | 1.5 天 |
| `llm-provider` 包的 compile 专用封装（temperature=0、重试、超时、prompt hash） | 1.5 天 |
| Compile golden eval set（见 § 10）          | 2 天   |
| 文档 + 例子                                 | 1 天   |
| **合计**                                  | **27 天 ≈ 5.5 周专注** |

这是 v0.1 三大块（confidence ledger ~21 天、compile ~27 天、vector+ask 另算）里最大的一块。

**排序建议**：confidence ledger 先做（compile 的 patch 要写 ledger 事件，是下游依赖）。compile 第二。vector + ask 最后——它们是 query-time 增强，没有 compile 提供的结构化知识，vector 检索也只是"更模糊的碎片检索"。

## 10. Compile Golden Eval

compile 的质量不能靠"看着对"。需要专门的 eval set，跟 v0.0 的 retrieval golden set 平行。

### 10.1 Eval set 格式

```yaml
# .akb/eval/compile-golden.yaml
version: "1.0"
items:
  - id: c001
    description: "新论文应被判定为 supersede 而非 extend"
    setup:
      existingPages: ["fixtures/gc-fixed-threshold.md"]
      newSource: "fixtures/fast2026-adaptive-gc.md"
    expect:
      relations:
        - { againstPage: "page_gc001", relation: "supersede" }
      mustCreatePage: true
      mustNotDeleteContent: true

  - id: c002
    description: "措辞不同但同义的笔记应被判定为 duplicate 或 merge，不能 new"
    setup:
      existingPages: ["fixtures/wear-leveling.md"]
      newSource: "fixtures/wear-leveling-reworded.md"
    expect:
      relations:
        - { againstPage: "page_wl001", relationIn: ["duplicate", "merge"] }
      mustNotCreatePage: true
```

### 10.2 评测维度

| 维度                     | 怎么测                                          |
| ---------------------- | -------------------------------------------- |
| **关系判定准确率**            | classify 阶段输出 vs golden 标注的 relation，算 accuracy |
| **不删内容**               | apply patch 后，原 page 的每个 verbatim chunk 仍可在 lineage 追到 |
| **lineage 完整性**        | 每个 derived chunk 的上游都能解析，无断链                  |
| **降级率**                | 跑 N 个 source，多少比例 degraded                    |
| **可复现性**               | 同一 source 跑两次（temperature=0），patch 的 changes 结构一致 |
| **矛盾召回**               | 故意构造含矛盾的 source 对，contradict/supersede 是否被识别  |
| **误报率**                | 构造无矛盾、无关联的 source，是否错误地产生 extend/merge        |

### 10.3 CI 集成

```yaml
# 每次改 compile 相关代码或 prompt 时跑
- name: Compile eval
  run: akb eval compile --set .akb/eval/compile-golden.yaml
```

回归阈值：关系判定准确率下降 > 8%、或不删内容/lineage 完整性出现任何 fail → CI fail。

## 11. 验证标准

实施 compile 之后必须能演示：

1. **基本编译**：ingest 一篇与现有 page 相关的新笔记，`akb compile` 产出 patch，patch 含至少一个非 `new` 的 change。
2. **不碰 markdown**：跑完 `akb compile`，`git status` 显示只有 `.akb/patches/` 下有新文件，`pages/` 完全没动。
3. **supersede 链**：ingest 一篇实质更新现有设计的论文，compile 判定 supersede，apply 后形成正确的 supersede 链（confidence ledger § 4 的验证场景能跑通）。
4. **矛盾标注**：ingest 一篇与现有 page 局部冲突的 source，compile 判定 contradict，apply 后原内容仍在、矛盾标注块被插入。
5. **lineage 正向**：对一个 derived page 跑 `akb lineage`，能一路追到 verbatim 的 source chunk 和行号。
6. **lineage 反向**：对一个 source 跑 `akb lineage --reverse`，列出它影响的所有 page。
7. **降级不阻塞**：断网 / 把 model 配成无效值，`akb compile` 退回纯拷贝 ingest，exit 0，compileMeta.degraded=true。
8. **可复现**：同一 source `akb compile` 跑两次，两个 patch 的 changes 结构（relation + targetPage + operation）完全一致。
9. **replay 一致**：`akb compile replay` 一个归档 patch，diff 结果在结构层面无差异。
10. **duplicate 不产内容 change**：ingest 一篇与现有 page 实质重复的笔记，compile 只产 `confidence_only` change，apply 后 markdown 不变但 ledger 多一条 `source_added`。
11. **eval gate**：故意把一个 supersede 场景的 fixture 改坏，`akb eval compile` 检测到关系判定准确率下降并 fail。

任何一条做不到，说明 compile 设计或实现有漏洞，先修再继续。

## 12. 一句话总结

**compile 是把产品名字里的 "Compiler" 兑现的那个命令**。它把 ingest 从"拷贝文件 + 建索引"升级成"理解新 source 并把它编译进现有知识体系"——发现关联、判定关系、生成合并、标注矛盾、建取代链，全部在 ingest time 做一次，全部产出可 review 的 patch，全部带可追溯的 lineage。

与 confidence ledger 的分工：ledger 管"知识会腐烂"（时间维度），compile 管"知识不会自己长在一起"（结构维度）。与 RAG 的分工：RAG 把组织工作放 query time 每次重做，compile 放 compile time 做一次、固化进 markdown。

v0.0 接通管道，confidence ledger 防腐烂，compile 让知识真正"编译"成一个整体。三块拼齐，才是 "AI-Native Knowledge Compiler" 这个名字真正成立的时候。
